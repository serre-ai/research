# Closed-Loop Experimentation -- Automated Hypothesis Testing

**Status:** Proposed
**Priority:** High
**Estimated effort:** 18-23 hours
**Dependencies:** Knowledge graph (not yet built), existing eval pipeline, PostgreSQL schema

## Problem

The eval pipeline is fire-and-forget. We design benchmarks in Python (`projects/reasoning-gaps/benchmarks/`), run them via `EvalJobManager`, dump results to PostgreSQL and JSONL, and manually inspect. There is no automated loop that turns findings into new hypotheses, designs follow-up experiments, runs them, and updates beliefs.

Concrete example: the B2 `budget_cot` anomaly (-0.254 lift, meaning CoT actively hurts performance) was discovered during analysis but requires manual intervention to investigate. The platform should autonomously design a follow-up experiment to understand why, run it within budget, analyze results, and update the knowledge graph.

## Architecture

### The Loop

```
1. Planner identifies testable hypothesis from knowledge graph
2. Experimenter designs minimal experiment specification
3. Budget check -- reject if experiment exceeds remaining daily budget
4. Experiment runs (eval pipeline via eval-bridge.ts)
5. Analyzer processes results, computes statistics
6. Knowledge graph updated: hypothesis confirmed / refuted / refined
7. Planner reassesses: what changed? what's next?
```

Each iteration is budget-bounded and produces a structured `ExperimentResult` that is both machine-readable (for the next loop iteration) and human-readable (for dashboard display and Slack alerts).

### Where This Fits

```
daemon.ts (cycle)
  ├── scoreProjects()          -- existing
  ├── runSession()             -- existing
  ├── evalManager.tick()       -- existing
  └── experimentLoop.tick()    -- NEW: runs one loop iteration per cycle
        ├── checkPendingResults()
        ├── analyzeCompletedExperiments()
        ├── updateBeliefs()
        └── designNextExperiment()
```

The experiment loop runs inside the daemon cycle, after eval jobs are ticked. It is not a separate service -- it extends the daemon's per-cycle work.

## New Interfaces

### ExperimentSpec

```typescript
interface ExperimentSpec {
  id: string;                          // UUID, auto-generated
  hypothesis: string;                  // "CoT helps more for depth-bounded tasks"
  hypothesisClaimId: string;           // reference to knowledge graph claim node
  task: string;                        // benchmark task ID (e.g. "B2_temporal_ordering")
  models: string[];                    // e.g. ["anthropic:claude-haiku-4-5", "openai:gpt-4o-mini"]
  conditions: string[];                // e.g. ["direct", "short_cot", "budget_cot"]
  parameters: Record<string, unknown>; // task-specific (difficulty range, sample distribution, etc.)
  sampleSize: number;                  // instances per model/condition combo
  successCriteria: string;             // natural language, interpreted by analyzer
  estimatedCost: number;               // USD, computed from pricing.ts
  maxCost: number;                     // hard cap -- abort if exceeded
  priority: "low" | "normal" | "high";
  status: "designed" | "approved" | "running" | "completed" | "failed" | "cancelled";
  createdAt: string;
  parentExperimentId?: string;         // if this is a follow-up
}
```

interface ExperimentResult {

## New Files

### `orchestrator/src/experiment-loop.ts`

The core class. Manages the full hypothesis-to-belief-update cycle.

```typescript
export class ExperimentLoop {
  constructor(
    rootDir: string,
    evalManager: EvalJobManager,
    budgetTracker: BudgetTracker,
    logger: ActivityLogger,
    notifier: Notifier,
    dbPool: pg.Pool | null,
  ) {}

  /**
   * Called every daemon cycle. Checks for completed experiments,
   * analyzes results, updates beliefs, designs follow-ups.
   */
  async tick(): Promise<void>;

  /**
   * LLM-assisted experiment design. Given a hypothesis (from knowledge graph
   * or from a previous experiment's follow-up suggestions), produces a
   * minimal ExperimentSpec.
   *
   * Uses Claude to:
   * - Determine which task and conditions test the hypothesis
   * - Estimate required sample size for statistical power
   * - Estimate cost using pricing.ts
   * - Set success criteria
   */
  async designExperiment(hypothesis: string, claimId?: string): Promise<ExperimentSpec>;

  /**
   * Submit an experiment to the eval pipeline.
   * Creates EvalJob entries via EvalJobManager for each model/condition combo.
   * Tracks mapping from ExperimentSpec.id -> EvalJob.id[].
   */
  async execute(spec: ExperimentSpec): Promise<void>;

  /**
   * After all eval jobs for an experiment complete, run statistical analysis.
   * Pulls results from PostgreSQL, computes CIs, effect sizes, p-values.
   * Then calls LLM to interpret results in context of the hypothesis.
   */
  async analyze(specId: string): Promise<ExperimentResult>;

  /**
   * Update knowledge graph based on experiment results.
   * - Adjust confidence on hypothesis claim
   * - Create new 'finding' claims with 'supports'/'contradicts' relations
   * - Mark experiment as integrated
   */
  async updateBeliefs(result: ExperimentResult): Promise<void>;

  /**
   * Generate follow-up experiment specs from results.
   * Called when analysis reveals anomalies or suggests refinements.
   * Follow-ups are queued, not immediately executed.
   */
  async suggestFollowUp(result: ExperimentResult): Promise<ExperimentSpec[]>;

  /**
   * List all experiments with optional status filter.
   */
  listExperiments(status?: ExperimentSpec["status"]): ExperimentSpec[];

  /**
   * Get experiment with its result.
   */
  getExperiment(id: string): { spec: ExperimentSpec; result?: ExperimentResult } | undefined;
}
```

### `orchestrator/src/eval-bridge.ts`

Bridges the TypeScript orchestrator with the Python eval pipeline. The existing `EvalJobManager` manages jobs via systemd units on the VPS. The eval bridge provides a higher-level interface for the experiment loop.

```typescript
export class EvalBridge {
  constructor(
    evalManager: EvalJobManager,
    dbPool: pg.Pool | null,
    benchmarkDir: string,  // default: projects/reasoning-gaps/benchmarks
  ) {}

  /**
   * Submit experiment as eval jobs.
   * For each model x condition combo, enqueues via EvalJobManager.
   * Returns job IDs for tracking.
   */
  async submitExperiment(spec: ExperimentSpec): Promise<string[]>;

  /**
   * Check if all jobs for an experiment have completed.
   */
  async isExperimentComplete(jobIds: string[]): Promise<boolean>;

  /**
   * Pull results from PostgreSQL for a specific experiment.
   * Queries eval_results table filtered by model, task, condition.
   * Returns structured data ready for statistical analysis.
   */
  async fetchResults(spec: ExperimentSpec): Promise<RawExperimentData[]>;

  /**
   * For local development (no systemd), spawn Python eval scripts directly.
   * Executes: python run_evaluation.py --model X --task Y --condition Z --sample-size N
   * Captures stdout/stderr, streams progress via callback.
   */
  async runLocal(
    spec: ExperimentSpec,
    onProgress?: (msg: string) => void,
  ): Promise<void>;
}

interface RawExperimentData {
  model: string;
  task: string;
  condition: string;
  instances: Array<{
    instanceId: string;
    correct: boolean;
    latencyMs: number;
    difficulty: number;
  }>;
}
```

### `orchestrator/src/stats.ts`

Statistical analysis utilities. Pure functions, no external dependencies beyond Node.js built-ins.

```typescript
/**
 * Compute 95% confidence interval for a proportion using Wilson score interval.
 * More accurate than normal approximation for small samples and extreme proportions.
 */
export function wilsonCI(successes: number, n: number, z?: number): [number, number];

/**
 * Compute Cohen's h effect size for two proportions.
 * Appropriate for comparing accuracy rates between conditions.
 */
export function cohenH(p1: number, p2: number): number;

/**
 * Two-proportion z-test.
 * Returns p-value for the null hypothesis that p1 = p2.
 */
export function proportionZTest(
  successes1: number, n1: number,
  successes2: number, n2: number,
): { zStat: number; pValue: number };

/**
 * Compute required sample size for detecting a given effect size
 * with specified power and significance level.
 */
export function requiredSampleSize(
  effectSize: number,
  power?: number,       // default 0.80
  alpha?: number,       // default 0.05
): number;

/**
 * Bootstrap confidence interval for effect size.
 * Used when parametric assumptions may not hold.
 */
export function bootstrapCI(
  data1: boolean[],
  data2: boolean[],
  nBootstrap?: number,  // default 10000
  alpha?: number,       // default 0.05
): { effectSize: number; ci: [number, number] };
```

## Integration Details

### Integration with EvalJobManager

The experiment loop does not replace `EvalJobManager` -- it sits on top of it. The relationship:

```
ExperimentLoop
  └── EvalBridge
        └── EvalJobManager  (existing, unchanged)
              └── systemd units / Python scripts
```

`ExperimentLoop.execute()` calls `EvalBridge.submitExperiment()`, which calls `EvalJobManager.enqueue()` for each model/condition combo. The experiment loop tracks which eval job IDs belong to which experiment spec.

### Integration with Daemon

The daemon gains one new call in its `cycle()` method:

```typescript
// In daemon.ts cycle(), after evalManager.tick():
try {
  await this.experimentLoop.tick();
} catch (err) {
  console.error("Experiment loop tick error:", err);
}
```

The experiment loop's `tick()` is idempotent and safe to call every cycle. It:
1. Checks if any running experiments have all their eval jobs completed
2. If so, runs analysis and belief updates
3. If the follow-up queue is non-empty and budget allows, designs and submits the next experiment

### Integration with Knowledge Graph

Each experiment spec references a `hypothesisClaimId`. When results come in:
- If hypothesis supported: increase confidence on the claim, add a `finding` claim with `supports` relation
- If contradicted: decrease confidence, add `finding` with `contradicts` relation
- If inconclusive: add `finding` with `refines` relation and note what additional evidence is needed

The knowledge graph schema (to be defined separately) needs:
- `claims` table: `id`, `type` (hypothesis | finding | observation), `statement`, `confidence`, `source`
- `claim_relations` table: `from_id`, `to_id`, `relation` (supports | contradicts | refines | extends)
- `claim_evidence` table: `claim_id`, `experiment_id`, `effect_size`, `p_value`

### Integration with BudgetTracker

Before executing any experiment, the loop calls `budgetTracker.canSpend(spec.estimatedCost)`. If the budget is insufficient:
- The experiment is queued with status `designed` (not `approved`)
- It will be retried next cycle when budget may have reset
- If `maxCost` exceeds daily budget entirely, it is flagged for human review

Cost estimation uses `pricing.ts` (already exists) to calculate per-model token costs multiplied by expected sample sizes.

### Integration with API

New endpoints in `orchestrator/src/routes/experiments.ts`:

| Method | Path | Description |
|--------|------|-------------|
| `GET` | `/api/experiments` | List all experiments, filterable by status |
| `GET` | `/api/experiments/:id` | Get experiment spec + result |
| `POST` | `/api/experiments` | Manually submit an experiment hypothesis |
| `POST` | `/api/experiments/:id/cancel` | Cancel a running experiment |
| `GET` | `/api/experiments/:id/progress` | Real-time progress (delegates to eval jobs) |

WebSocket channel: `experiments` -- broadcasts experiment state changes.

### Integration with Notifier

Events that trigger Slack notifications:
- Experiment completed with results summary
- Hypothesis confirmed or refuted (high-signal event)
- Anomaly detected in results
- Follow-up experiment auto-designed
- Budget exceeded during experiment

## Database Schema Additions

```sql
-- Experiment specifications
CREATE TABLE experiment_specs (
    id              TEXT PRIMARY KEY,
    hypothesis      TEXT NOT NULL,
    claim_id        TEXT,                -- FK to future claims table
    task            TEXT NOT NULL,
    models          JSONB NOT NULL,      -- string array
    conditions      JSONB NOT NULL,      -- string array
    parameters      JSONB DEFAULT '{}',
    sample_size     INTEGER NOT NULL,
    success_criteria TEXT,
    estimated_cost  REAL,
    max_cost        REAL,
    priority        TEXT DEFAULT 'normal',
    status          TEXT NOT NULL DEFAULT 'designed',
    parent_id       TEXT REFERENCES experiment_specs(id),
    created_at      TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    completed_at    TIMESTAMPTZ
);

-- Experiment results (one row per completed experiment)
CREATE TABLE experiment_results (
    spec_id             TEXT PRIMARY KEY REFERENCES experiment_specs(id),
    hypothesis_supported BOOLEAN,        -- null = inconclusive
    confidence          REAL,            -- p-value
    effect_size         REAL,
    effect_size_ci      JSONB,           -- [lower, upper]
    summary             TEXT,
    follow_up_hypotheses JSONB,          -- string array
    anomalies           JSONB,           -- string array
    cost_usd            REAL,
    completed_at        TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

-- Maps experiment specs to eval jobs
CREATE TABLE experiment_jobs (
    spec_id     TEXT NOT NULL REFERENCES experiment_specs(id),
    job_id      TEXT NOT NULL,
    model       TEXT NOT NULL,
    condition   TEXT NOT NULL,
    PRIMARY KEY (spec_id, job_id)
);
```

## Validation: B2 budget_cot Anomaly

The first real test of this system. The loop should:

1. **Planner observes**: B2 `budget_cot` lift is -0.254 (CoT hurts). This contradicts the general finding that CoT helps for depth-bounded tasks.
2. **Hypothesis generated**: "budget_cot's negative effect on B2 is caused by the budget prompt encouraging premature truncation of temporal reasoning chains"
3. **Experiment designed**: Run B2 with `budget_cot` across difficulty levels 1-5, sample size 100/level, on 3 models. Estimated cost: ~$3.
4. **Experiment runs**: 15 eval jobs (3 models x 5 difficulty levels) submitted via `EvalBridge`
5. **Analysis**: Compare accuracy by difficulty. If degradation concentrates at high difficulty, the truncation hypothesis is supported.
6. **Belief update**: Adjust confidence on "CoT universally helps depth-bounded tasks" and create a refined claim about budget-constrained CoT.
7. **Follow-up**: If supported, design experiment testing alternative budget prompts that preserve reasoning depth.

Expected cost for this full investigation: $5-10 across 2-3 loop iterations.

## Success Criteria

- The B2 anomaly can be investigated autonomously: system designs follow-up, runs it, analyzes, updates knowledge graph -- no human intervention required.
- New hypotheses are generated from unexpected results (not just confirmations).
- Cost per experiment-loop iteration stays under $5 for typical single-task experiments.
- Results are queryable through the API and visible on the dashboard.
- Budget guardrails prevent runaway spending: no single experiment exceeds `maxCost`, no day exceeds `dailyBudgetUsd`.

## Open Questions

1. **Knowledge graph dependency**: The experiment loop references claim IDs, but the knowledge graph does not exist yet. Should we build a minimal claims table first, or let the experiment loop work without it initially (storing hypothesis text only)?
   - **Recommendation**: Build minimal claims table (3 columns: id, statement, confidence) as part of this work. Full knowledge graph is a separate roadmap item.

2. **LLM for experiment design**: Which model should design experiments? Using Sonnet 4 keeps costs low (~$0.05/design) but Opus 4 may produce better experimental designs.
   - **Recommendation**: Sonnet 4 by default, with an option to escalate to Opus 4 for high-priority hypotheses.

3. **Approval gate**: Should experiments above a cost threshold require human approval, or run fully autonomously?
   - **Recommendation**: Fully autonomous below $10. Experiments estimated above $10 are flagged in Slack and held in `designed` status until the next daemon cycle (giving the human time to cancel if needed).
